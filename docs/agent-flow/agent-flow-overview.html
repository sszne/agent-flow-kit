Plan first, evidence last

Agent Flow は、判断・実装・検証を同じ証拠でつなぐ作業導線です。

コードを書く前に業務フロー、回帰面、検証契約を固定し、実装後は証拠とレビューで閉じます。 AI に任せる範囲を広げつつ、人が決めるべき意思決定を見失わないためのキットです。

この HTML の見方

上段は導入からレビューまでの流れです。赤いノードはゲート、黄色いノードは分岐判断、 緑のノードは後続作業の根拠になる成果物を表します。

Overall map

全体フロー図

Agent Flow は、初回オンボーディングでプロジェクト知識を作り、以後の変更では `flow-plan` を中心に計画、実装、統合証拠、レビューを接続します。

1. Knowledge base 導入・棚卸し

AFK-010 flow-document

要件資料や提案資料を候補証拠として取り込み、確認済み・矛盾・未来範囲・要確認に分類します。

AFK-002 project-structure-survey

実ソース、ルート、スキーマ、テスト、デプロイ情報から構造・入口・運用リスクを記録します。

AFK-003 business-flow-discovery

業務フロー、通常系、例外、権限、境界、side effect、回帰面を行列化します。

AFK-004 integration-scenario-design

各フローを Unit、Feature/API、Playwright、runtime/provider、manual blocker に割り当てます。

2. Planning gates 判断・計画

START flow-start or flow-plan

flow-start は新規発見、flow-plan は既存挙動変更・バグ・回帰・業務フロー変更で使います。

GATE Goal and questioning

目的、成功条件、根本原因対象、質問の要否を明示し、推測だけで計画を凍結しないようにします。

GATE Risk-specific checks

runtime causality、bug feedback、frontend design-system、provider/auth/deploy 証拠レーンを必要時に走らせます。

PLAN Frozen plan

Business Flow Matrix、Regression Surface Matrix、Test Design Matrix、Integration Coverage Contract を含む計画を保存します。

REVIEW? Plan Review Requirement

大規模・高影響なら flow-plan-review 必須。小さな局所変更なら理由付きで任意にできます。

3. Execution loop 実装・証拠・レビュー

AFK-006 flow-plan-review

高影響計画を実装前に別視点で確認し、見落とした DB、権限、外部依存、テストリスクを検出します。

AFK-007 flow-impl / team-implement

凍結計画のタスク順に実装し、必要なら TDD の Red から始め、実装レポートを更新します。

AFK-008 flow-integration-test

画面や複数ステップの業務フローは Playwright 証拠、スクリーンショット、影響レビューで検証します。

REVIEW team-review

実差分をもとに security、quality、test coverage を確認し、残リスクと未検証面を明確にします。

agent-router

危険そうな依頼を `/flow-plan` へ誘導し、既存挙動変更の直実装を抑えます。

flow-plan-required-gate

挙動影響パスの編集に凍結計画を要求します。表示のみの軽微変更は狭く通します。

matrix gate / CI

必須行列、レビュー要否、曖昧な waiver、オンボーディング文書の欠落を検出します。

installer dry-run

portable assets と target-local guidance を分類し、ローカル判断の上書きを避けます。

Solved problems

具体的に解決できる課題

Agent Flow は「AI に実装させる」だけの仕組みではなく、計画不足・検証不足・証拠不足を 実装前後のゲートで減らすための仕組みです。

01

隣接業務フローの見落とし

Business Flow Matrix と Regression Surface Matrix により、編集ファイルではなく影響する業務フローからテストを設計できます。

02

曖昧な依頼のまま実装開始

Goal Confirmation と Questioning Decision により、目的・成功条件・質問スキップ理由を計画に残します。

03

古い資料の真実化

flow-document が資料を sidecar 証拠として扱い、未確認・矛盾・未来範囲の主張が実装要件へ混入するのを防ぎます。

04

本番・外部要因の誤診断

Runtime Causality Gate がデプロイ、provider、secret、binding、remote data、auth/session をコード変更前に分類します。

05

テストが関数単位だけに閉じる

Integration Coverage Contract が happy、exception、permission、boundary、side-effect、regression を明示的に要求します。

06

レビュー過多またはレビュー不足

Plan Review Requirement により、高影響変更はレビュー必須、小さな局所変更は理由付き任意にできます。

07

画面検証の証拠不足

表示・複数ステップの業務フローは Playwright 証拠、index.html、test review、business-flow impact に集約します。

08

キット更新時のローカル知識破壊

installer dry-run が safe update と preserve local を分類し、対象リポジトリ固有のルールや設定を手動マージ対象にします。

Human decision points

人力による判断・指示が必要な箇所

Agent Flow は AI が調査、整理、提案、実装、検証証拠の作成を担えるようにします。 ただし、責任ある判断や外部権限が必要な部分は人が明示する必要があります。

箇所 AI が準備できること 人が判断・指示すること
資料インテーク 資料を読み、主張を分類し、リポジトリ証拠と矛盾する点を抽出します。 どの資料を使うか、未確認 claim を採用するか、古い資料を無視するかを決めます。
目的と成功条件 現象、原因候補、実装候補、検証候補を整理して質問します。 症状表示で十分か、根本原因除去が必要か、完了とみなす証拠を指定します。
業務ルール 既存コード、DB、テスト、画面から actor、権限、状態遷移、side effect を推定します。 actor、scope、権限、データ所有、例外時の業務判断など、コードだけでは決められないルールを確定します。
設計・依存関係 既存パターン、最小実装、代替案、リスク、テスト容易性を比較します。 新規ライブラリ、外部サービス、運用コスト、セキュリティ許容度などの選択を承認します。
レビュー要否 高影響条件に照らして Required / Optional の候補理由を書きます。 不確実性が高い変更、公開 API、認証、schema、外部連携などを高影響扱いにするか最終判断します。
外部・本番証拠 local mock、deployed artifact、provider/device、valid session の証拠レーンを分けて計画します。 認証情報、production/sandbox 操作、実端末確認、外部 provider 操作の実行または許可を行います。
waiver / blocker 未検証面と理由を具体化し、残リスクとして報告します。 ブロックを許容して進めるか、追加検証を要求するか、リリースを止めるかを判断します。
最終受け入れ 実装内容、検証結果、証拠パス、残リスクをまとめます。 業務上の受け入れ、デプロイ、公開、運用告知、フォローアップの優先度を決めます。

Important

Agent Flow は「人の判断を不要にする」仕組みではありません。AI が証拠を集め、判断材料を構造化し、 実装・検証を進めます。人は目的、業務判断、外部権限、許容リスクを明示することで、AI の作業を安全に速くできます。
